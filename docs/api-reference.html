API-Referenz

Vollständige Referenz aller AdminHelper Server API-Endpunkte. Basis-URL: https://<server>:443/api

Authentifizierung Verbindungen

1) Authentifizierung

Die API unterstützt zwei Authentifizierungsmethoden:

JWT Bearer Token

Login über POST /api/auth/login liefert ein access_token (Gültigkeit: 8 h) und ein refresh_token (Gültigkeit: 7 d). Das Access-Token wird als Bearer-Token im Header übergeben. 

# Login und Token erhalten
curl -X POST https://server:443/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "geheim"}'

# Token verwenden
curl https://server:443/api/connections \
  -H "Authorization: Bearer <access_token>"

API-Key

API-Keys werden über die Web-UI oder die API erstellt. Der Key wird im Header X-API-Key übergeben. Berechtigungen: read oder read_write. 

# API-Key verwenden
curl https://server:443/api/connections \
  -H "X-API-Key: <api_key>"
Auth-Level in den folgenden Tabellen:
  • Public — Kein Token erforderlich
  • JWT — Bearer Token erforderlich
  • Admin — Bearer Token + Admin-Flag
  • API-Key — API-Key Header (read oder read_write)
  • JWT|API-Key — Beides akzeptiert

12) Hooks

Hooks — Webhooks, Events und Schedules
Method Pfad Auth Beschreibung
GET /api/hooks Admin Alle Hooks auflisten
POST /api/hooks Admin Hook erstellen (webhook/event/schedule)
PUT /api/hooks/{id} Admin Hook aktualisieren
DELETE /api/hooks/{id} Admin Hook löschen
POST /api/hooks/{id}/trigger Admin Hook manuell auslösen
POST /api/hooks/incoming/{token} Public Webhook-Endpoint (via Token)

Hook-Typen

  • Webhook: HTTP POST mit Token-Authentifizierung, rate-limited (20/min).
  • Event: Automatisch bei Systemereignissen (connection.created, server.updated, frp.config.created etc.).
  • Schedule: Cron-basiert, z.B. 0 2 * * * = täglich 02:00.
