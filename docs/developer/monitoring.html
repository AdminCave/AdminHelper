Monitoring-Stack

Monitoring ist ein Zusammenspiel aus drei Ebenen: SRM-Server definiert Checks und aggregiert Alerts, der separate monitoring/-Service führt Check-Logik aus und schreibt Metriken, VictoriaMetrics speichert die Zeitreihen.

Daten-Fluss

flow
[Agent]                [Monitoring-Service]       [VictoriaMetrics]
    │                         │                            │
    │ POST /ingest ─ metrics ▶│                            │
    │                         │ /api/v1/import/prometheus ▶│
    │                         │                            │
                              │ ◀ pull Config from /api/monitoring
                              │
   [SRM-Server] ─ schedule ─▶ APScheduler triggers checks ─▶ [Monitoring-Service]
                              │
                              │ SRM ◀ query Charts ◀ uPlot (Web-Panel)

SRM-Server-Komponenten

  • server/app/monitoring/checks.py – Pydantic-Schemas pro Check-Typ
  • server/app/monitoring/scheduler.py – APScheduler-Integration
  • server/app/monitoring/proxy.py – SSRF-sicherer Proxy zu VictoriaMetrics
  • server/app/monitoring/alerts.py – Threshold-Evaluation + Hook-Trigger

Monitoring-Service

Eigenständige FastAPI-App unter monitoring/. Lauscht auf Port 8080 (container-intern), öffentlich 8480 für Agent-Ingest. Authentifiziert mit MONITOR_API_KEY.

Check-Typ-Schemata

Jeder Check-Typ hat ein Pydantic-Schema für Config und ein Executor. Beispiel http:

python
class HttpCheckConfig(BaseModel):
    url: HttpUrl
    method: Literal["GET", "HEAD", "POST"] = "GET"
    expected_codes: list[int] = [200]
    timeout_sec: float = 5.0
    verify_tls: bool = True
    headers: dict[str, str] | None = None

Executor ruft httpx.AsyncClient auf, misst Latenz, erzeugt ein CheckResult und pusht Metriken an VictoriaMetrics.

SSRF-Schutz

Der Monitoring-Proxy im Server darf VictoriaMetrics-Queries an das Web-Panel durchreichen, aber die Query-URL wird strikt allowlisted: /api/v1/query und /api/v1/query_range. Raw-Host-Zugriffe sind blockiert, damit keine internen Dienste erreichbar sind.

Ingest-Validierung

Der Ingest-Endpunkt prüft, dass Samples aus dem Agent wirklich zur eigenen Server-ID gehören. Ein kompromittierter Agent-Key kann also keine fremden Server-Metriken fälschen.

Lokales Debugging

bash
# Einzelnen Check ausführen, Debug-Output an stdout
docker compose exec monitoring python -m app.cli run-check http 

# VictoriaMetrics direkt fragen
docker compose exec monitoring curl -s \
    "http://victoria:8428/api/v1/query?query=cpu_load{server=\"srv1\"}"
