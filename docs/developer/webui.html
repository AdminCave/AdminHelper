Web-Panel

Das Web-Panel ist eine eigenständige Svelte-5-SPA mit Hash-Router. Es liegt unter frontend-src/, wird via Vite gebaut und im Docker-Image per Multi-Stage in das Python-Runtime-Image kopiert.

Projekt-Struktur

tree
frontend-src/
├── src/
│   ├── lib/
│   │   ├── api/           fetch-Wrapper für /api/*
│   │   ├── components/    Button, Modal, TagChip, Tabs, ...
│   │   ├── stores/        13 Runes-Stores (auth, connections, ...)
│   │   ├── i18n/          DE/EN-Dictionaries
│   │   └── utils/
│   ├── routes/            Pages (hash-routed)
│   ├── App.svelte
│   └── main.ts
├── tests/e2e/             Playwright
├── svelte.config.ts
└── vite.config.ts

Routing

Hash-Router (kein SSR, kein Client-Router von SvelteKit): Die Pfade sind #/connections, #/servers usw. Das funktioniert zuverlässig hinter beliebigen Reverse-Proxies, ohne dass der Server SPA-Fallbacks verstehen muss. Implementierung in src/lib/router.ts.

Stores

Svelte-5-Runes ersetzen die klassischen Writables. Jeder Store ist eine TypeScript-Klasse mit $state-Feldern und Methoden:

StoreZweck
authStoreLogin, Logout, Refresh, aktueller User.
connectionsStoreLive-Liste + Suche + Gruppenfilter.
serversStoreServer-Inventar + Provision-Token.
apiKeysStoreKey-CRUD.
hooksStoreHook-CRUD + Execution-Log.
monitoringStoreChecks, Alerts, VictoriaMetrics-Queries.
ansibleStorePlaybook-Läufe, Streaming-Output.
tunnelStoreFRP-Tunnel-Status und -CRUD.
toastStoreGlobale Toast-Benachrichtigungen.
confirmStoreBestätigungs-Dialoge.
languageStoreDE/EN-Umschaltung (i18n).
themeStoreKein Theme-Switch – immer Dark (aus CSS-Tokens).
settingsStoreServerseitige Settings.

API-Wrapper

Alle HTTP-Calls gehen durch src/lib/api/client.ts. Der Client verwaltet Access-Token und triggert automatisch Refresh, wenn ein 401 zurückkommt. Bei zweitem 401 wird zur Login-Page geleitet.

uPlot-Charts

Monitoring-Charts nutzen uplot direkt – kein schwergewichtiges Chart-Framework. Wrapper in src/lib/components/charts/. uPlot ist 30 kB gzipped und rendert schnell auch 100k Punkte.

Tests

Playwright-Smoketests deck alle 8 Produktiv-Seiten ab: Login, Connections, Servers, Users, API-Keys, Hooks, Ansible, FRP-Tunnels, Monitoring. Visual-Diff-Screenshots werden als CI-Artifact hinterlegt.

bash
cd frontend-src
npm run check           # svelte-check + tsc
npm run lint            # eslint + svelte-eslint
npx playwright test     # E2E

Build & Deployment

Der Vite-Build landet in frontend-src/dist/. Im Dockerfile wird dieser Output in /app/frontend/ kopiert, und die FastAPI-App mountet ihn als Static-Route. SPA-Fallback prüft vorher Datei-Existenz, bevor index.html zurückgegeben wird – das verhindert MIME-Fehler auf /assets/*.js.
