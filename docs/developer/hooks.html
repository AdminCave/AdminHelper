Hooks & Events

Das Hook-System ist der Haupt-Integrationspunkt nach außen. Jedes relevante Ereignis im Server feuert ein Event, registrierte Scripts laufen in einer sanitisierten Subprozess-Umgebung mit harter Timeout-Grenze.

Architektur

tree
apps/server/app/modules/hooks/
├── router.py          /api/hooks CRUD
├── models.py          Hook
├── schemas.py         Pydantic-Schemas
├── scheduler.py       Schedule-Hook-Cron + Notification-Outbox
├── script_runner.py   Subprocess-Spawn (async)
└── script_worker.py   Child-Prozess: stdin=JSON, stdout/stderr capture

Event-Katalog

EventPayload (Auszug)
connection.created / updated / deletedConnection-Objekt
connections.importedcount, mode
server.created / updated / deletedid, name (bei created zusätzlich hostname)
server.startup(kein event_data)
user.created / deletedid, username (bei created zusätzlich is_admin)
frp.config.created / updated / deletedid, name
frp.tunnel.created / updated / deletedid, name (bei created zusätzlich serverId)
playbook.created / updated / deletedid, name
alert.triggeredserver_id, severity, title, message — feuert bei einer kritischen Monitoring-Transition (monitoring.check.transition mit Severity critical)

Hook-Definition

json
{
  "name": "Slack-Alert",
  "hook_type": "event",
  "event_triggers": ["server.updated"],
  "script": "http_post('https://hooks.slack.com/services/T000/B000/XXX', json_data={'text': f'AdminHelper: {event_type}'})\nresult['notified'] = True"
}

Das script-Feld ist Python (kein Shell), das der Worker per exec() ausführt. Die Context-Keys des Events stehen als Variablen bereit (bei event-Hooks event_type und event_data, bei schedule triggered_at/last_run, bei webhook payload/headers/params); Ausgaben laufen über das result-Dict und log(...). Aus Sicherheitsgründen ist headers auf eine Safe-Liste beschränkt (content-type, user-agent, x-hook-source) — client-gesetzte Header wie Authorization, Cookie und X-Forwarded-* gelangen nie ins Script. Der gesamte Context wird als JSON an den Worker übergeben (json.dumps mit default=str), daher überleben nur JSON-native Typen — nicht-native Werte in event_data (datetime, Decimal, bytes, ORM-Objekte) erreichen das Script als ihre str()-Form, nicht als Originaltyp.

Environment-Modell

Hook-Scripts laufen in einem eigenen Python-Subprozess mit reduziertem Environment — das ist Crash-/Timeout-Isolation, kein Security-Sandbox:

  • Reduziertes Env: Nur PATH, PYTHONPATH, DATABASE_URL und DATA_DIR werden durchgereicht (ADMIN_PASSWORD, MONITOR_API_KEY, REDIS_URL fallen weg). SECRET_KEY und die DB-Zugangsdaten bleiben aber erreichbar (über DATA_DIR/.secret_key bzw. DATABASE_URL) — kein Secret-Jail.
  • CWD ist das Server-Verzeichnis (server_dir), kein Temp-Verzeichnis.
  • Timeout über subprocess.run(timeout=…), Default 30 s; beim Ablauf wird der Worker-Prozess gekillt, nicht zwingend dessen Kindprozesse.
  • Kein Output-Limit und kein User-Switch: stdout/stderr werden vollständig eingesammelt, der Worker läuft als derselbe Benutzer wie der Server.
Kein vollständiger Sandbox

Die Hook-Isolation ist Defense-in-Depth, nicht ein Seccomp/Namespace-Jail. Admins mit Hook-CRUD-Rechten können beliebigen Code auf dem Server-Host ausführen. Hook-Zugriff daher nur vertrauenswürdigen Rollen geben.

Fehlerverhalten

Schlägt ein Hook fehl (exit_code != 0 oder Timeout), wird der Fehler protokolliert: Event- und Schedule-Hooks per logger.exception im Server-Log, der Webhook-Trigger zusätzlich im Response-Body (success: false, error). Einen automatischen Retry oder ein hook.failed-Meta-Event gibt es aktuell nicht; ein fehlgeschlagener Hook blockiert weder andere Hooks noch den auslösenden Request.

Script-Runner-Implementierung

python
SCRIPT_TIMEOUT_SECONDS = 30

def run_hook_script(script, hook_type, context, timeout=SCRIPT_TIMEOUT_SECONDS):
    payload = json.dumps({"script": script, "context": context}, default=str)

    # Reduziertes Env: entfernt ADMIN_PASSWORD, MONITOR_API_KEY, REDIS_URL. KEIN
    # Jail — DATABASE_URL ist bewusst vererbt (Hooks brauchen DB), SECRET_KEY bleibt
    # ueber DATA_DIR/.secret_key erreichbar. Hooks = vertrauenswuerdiger Code.
    server_dir = str(Path(__file__).parents[3])
    worker_env = {
        "PATH": os.environ.get("PATH", ""),
        "PYTHONPATH": server_dir,
        "DATABASE_URL": os.environ.get("DATABASE_URL", ""),
        "DATA_DIR": os.environ.get("DATA_DIR", ""),
    }
    try:
        proc = subprocess.run(
            [sys.executable, _WORKER_SCRIPT],  # fuehrt das Script via Python-Worker aus
            input=payload, capture_output=True, text=True,
            timeout=timeout, cwd=server_dir, env=worker_env,
        )
    except subprocess.TimeoutExpired:
        return {"success": False, "error": f"Timeout nach {timeout}s"}
    return _parse_worker_output(proc)  # success/result/logs aus stdout

Eigene Events feuern

Interner Code kann jederzeit Events feuern:

python
from app.core.events import fire_event

fire_event("server.updated", {"id": server.id, "name": server.name})

fire_event ist non-blocking: Der Dispatch läuft im Thread-Pool, der Aufruf kehrt sofort zurück. Die HTTP-Response des Auslösers wartet also nicht auf den Hook.

Test-Execution

Im Web-Panel kann jeder Hook mit einem Beispiel-Event geprüft werden: POST /api/hooks/{id}/test. Der Execution-Log zeigt stdout/stderr/exit_code in Echtzeit – praktisch beim Script-Debugging.
