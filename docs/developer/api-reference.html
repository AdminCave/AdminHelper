API-Referenz

Alle HTTP-Endpoints des SRM-Servers. Basis-URL: https://<host>/api. Vollständiges OpenAPI-Schema unter /api/docs (Swagger UI) bzw. /api/openapi.json.

Authentifizierung

Zwei Mechanismen, je nach Aufrufer:

  • JWT-Bearer (Web-Panel, Desktop): Authorization: Bearer <access_token>. Refresh per httpOnly-Cookie.
  • API-Key (Agents, Extension, Drittanbieter): ?api_key=srm_… als Query-Parameter oder X-API-Key-Header.

Auth

POST/api/auth/login

Body: { "username": "admin", "password": "…" }. Antwort: { "access_token": "…", "user": {…} }. Setzt refresh_token-Cookie.

POST/api/auth/refresh

Liest Refresh-Cookie. Rotiert den Refresh-Token, blacklisted den alten jti. Reuse eines geblacklisteten Tokens invalidiert alle Sessions des Users.

POST/api/auth/logout

Blacklisted den aktuellen Refresh-Token, löscht das Cookie.

GET/api/auth/me

Aktueller User inkl. roles, mfa_enabled.

Users

GET/api/users

Liste aller User (Admin-only).

POST/api/users

Neuen User anlegen. Passwort wird per Argon2id gehasht.

PATCH/api/users/{id}

Rollen, E-Mail oder is_active ändern.

DELETE/api/users/{id}

Soft-Delete: is_active = false, Sessions invalidiert.

Connections

GET/api/connections

Query: ?kind=rdp|ssh|vnc|web, ?group=…, ?search=….

POST/api/connections

Body: { kind, name, host, port, tags, tunnel_id? }.

PUT/api/connections/{id}

Vollständiger Replace.

DELETE/api/connections/{id}

Servers

GET/api/servers

Inventar, gruppiert nach status.

POST/api/servers/{id}/token

24h-Provision-Token für den Agent. Einmal einlösbar.

POST/api/servers/provision

Agent löst Token ein → Zertifikat + Config.

API-Keys

GET/api/apikeys
POST/api/apikeys

Body: { name, scope: [...], expires_at? }. Klartext-Key wird einmalig in der Response zurückgegeben, gespeichert wird nur SHA-256.

DELETE/api/apikeys/{id}

Hooks

GET/api/hooks
POST/api/hooks

Body: { event, script, timeout_sec, enabled }.

GET/api/hooks/{id}/executions

Letzte 50 Ausführungen mit exit_code, stdout, stderr.

POST/api/hooks/{id}/test

Ausführung mit Test-Event-Payload.

Monitoring

GET/api/monitoring/checks
POST/api/monitoring/checks

Body: { kind, server_id, config: {…}, threshold: {…}, interval_sec }.

POST/api/monitoring/ingest

Agent-Push. Body: Prometheus-Remote-Write-Payload. Auth: X-API-Key mit Scope monitor_ingest.

GET/api/monitoring/query

SSRF-sicherer Proxy zu VictoriaMetrics. Nur /api/v1/query und /api/v1/query_range erlaubt.

Alerts

GET/api/alerts

Query: ?state=firing|acknowledged|resolved.

POST/api/alerts/{id}/ack

Alert quittieren, löst alert.acknowledged-Hook aus.

FRP

GET/api/frp/tunnels
POST/api/frp/tunnels

Body: { name, type: "tcp"|"http"|"stcp", local_port, sk? }. stcp erwartet sk (Shared-Key).

POST/api/frp/client-cert/rotate

Agent-initiiert, Shortcut zu /api/servers/{id}/cert.

Audit

GET/api/audit

Query: ?since=ISO, ?user_id=, ?action=. Paginate via ?limit+?cursor.

Settings & Health

GET/api/settings

Globale Einstellungen (Branding, Timeouts, Feature-Flags).

GET/api/health

Liefert { status, version, git_commit, db: "ok", redis: "ok" }. Kein Auth.

Fehler-Format

json
HTTP/1.1 422 Unprocessable Entity
Content-Type: application/json

{
  "detail": [
    { "loc": ["body", "port"], "msg": "ensure this value is less than 65536", "type": "value_error" }
  ]
}

Fehler folgen Pydantic-Validations-Konventionen; 500-Responses enthalten nur { "detail": "internal error", "request_id": "…" } – Request-ID ist korrelierbar mit dem Server-Log.

Rate-Limits

  • Login: 5 Fehlversuche pro 15 min / IP (Redis).
  • Refresh: 20 req/min / Account.
  • Ingest: unlimitiert bei gültigem Agent-Key, aber Samples werden serverseitig thinned (Dedup pro metric,labels,ts).
Swagger direkt nutzen

/api/docs rendert OpenAPI 3.1 interaktiv, inklusive „Try it out". Für CI-Clients reicht ein openapi-generator generate -g go -i https://srv/api/openapi.json.
