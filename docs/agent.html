AdminHelper Agent — Referenz

Der AdminHelper Agent ist ein einheitlicher Go-Binary für FRP-Tunnel-Synchronisation und System-Monitoring auf Linux und Windows.

Installation Kommando-Referenz

1) Übersicht

Der adminhelper-agent ist ein einheitlicher Go-Binary, der die alten Shell- und Python-Scripts vollständig ersetzt. Er vereint zwei Hauptfunktionen in einem einzigen Programm:

FRP-Client-Synchronisation

Automatischer Abgleich der FRP-Client-Konfiguration mit dem AdminHelper-Server. Der Agent lädt Tunnel-Definitionen, PKI-Zertifikate und generiert die frpc.toml automatisch.

System-Monitoring Push

Sammelt Systemmetriken (CPU, RAM, Disk, Services, Temperaturen) und sendet diese per HTTPS-Push an den Monitoring-Service. Keine eingehenden Ports erforderlich.

Der Agent läuft als systemd-Timer (Linux) oder Windows-Service und führt alle 5 Minuten einen Synchronisations- und Monitoring-Zyklus durch. Unterstützte Plattformen: Linux amd64 und Windows amd64.

2) Installation

DEB-Paket (Debian/Ubuntu)

sudo apt install ./adminhelper-agent_0.13.0_amd64.deb

Installiert den Agent nach /usr/local/bin/ und legt systemd-Unit-Dateien an.

RPM-Paket (RHEL/AlmaLinux)

sudo rpm -i adminhelper-agent-0.13.0-1.x86_64.rpm

Analog zum DEB-Paket — Agent-Binary und systemd-Units werden installiert.

Windows Binary

Die EXE-Datei nach C:\Program Files\AdminHelper\ kopieren und als Windows-Service registrieren: 

copy adminhelper-agent.exe "C:\Program Files\AdminHelper\"
"C:\Program Files\AdminHelper\adminhelper-agent.exe" service install

3) Kommando-Referenz

Kommando Beschreibung
adminhelper-agent version Version anzeigen
adminhelper-agent run Daemon-Modus: FRP-Sync + Monitoring alle 5 Minuten
adminhelper-agent run --once Einmalige Ausführung (für systemd-Timer)
adminhelper-agent frpc init Ersteinrichtung FRP-Client. Flags: --url URL --token TOKEN --server-id ID [--cacert PATH] [--insecure]
adminhelper-agent frpc sync Einmalige Konfigurationssynchronisation
adminhelper-agent monitor init Ersteinrichtung Monitoring. Flags: --url URL --api-key KEY --server-id ID [--services svc1,svc2] [--cacert PATH] [--insecure]
adminhelper-agent monitor push Einmaliger Metrik-Push
adminhelper-agent service install Systemd-Timer oder Windows-Service registrieren
adminhelper-agent service uninstall Service deregistrieren

4) Konfigurationsdateien

Beide Konfigurationsdateien verwenden ein einfaches KEY=VALUE-Format (eine Variable pro Zeile).

FRP-Konfiguration (adminhelper.conf)

  • Linux: /etc/frp/adminhelper.conf
  • Windows: %ProgramData%\AdminHelper\frp\adminhelper.conf
# /etc/frp/adminhelper.conf
ADMINHELPER_URL=https://admin.example.com
API_KEY=abcdef1234567890abcdef1234567890
SERVER_ID=srv-web-prod-01
CACERT=/etc/frp/ca.crt
INSECURE=false

Monitor-Konfiguration (monitor.conf)

  • Linux: /etc/adminhelper/monitor.conf
  • Windows: %ProgramData%\AdminHelper\monitor.conf
# /etc/adminhelper/monitor.conf
MONITOR_URL=https://admin.example.com/api/monitoring
API_KEY=abcdef1234567890abcdef1234567890
SERVER_ID=srv-web-prod-01
SERVICES=nginx,postgresql,redis
CACERT=/etc/adminhelper/ca.crt
INSECURE=false
Hinweis: Die Konfigurationsdateien werden automatisch durch frpc init bzw. monitor init erstellt. Manuelle Bearbeitung ist nur für Sonderfälle notwendig.

5) Provisionierungs-Flow

  1. Server im AdminHelper Web-UI anlegen.
  2. Provision-Token generieren (Web-UI oder API).
  3. Auf dem Zielserver den Agent initialisieren: 
    sudo adminhelper-agent frpc init \
  --url https://server:443 \
  --token TOKEN \
  --server-id SERVER_ID
  4. Agent lädt PKI-Bundle herunter (CA-Zertifikat, Client-Zertifikat, Private Key).
  5. Agent erstellt adminhelper.conf und frpc.toml.
  6. Timer/Service starten: 
    sudo adminhelper-agent service install
  7. Automatische Synchronisation läuft ab sofort alle 5 Minuten.
Tipp: Monitoring kann im selben Schritt eingerichtet werden — einfach zusätzlich monitor init ausführen, bevor service install aufgerufen wird.

6) Monitoring-Collector

Der integrierte Collector erfasst folgende Systemmetriken bei jedem Push-Zyklus:

Metrik Details
CPU Auslastung in Prozent (1-Sekunden-Durchschnitt)
Memory Belegung in Prozent, Total MB, Used MB
Load 1m, 5m, 15m Durchschnitt (nur Linux)
Disks Pro Partition: Mountpoint, Dateisystem, Belegung %, Total/Used GB. Filtert automatisch: squashfs, tmpfs, devtmpfs, overlay, proc, sysfs
Uptime Systemlaufzeit in Sekunden
Services Status konfigurierter Dienste (systemd auf Linux, Windows Services)
Temperatures CPU-Temperatursensoren (wenn verfügbar)

7) Systemd-Integration

Auf Linux nutzt der Agent eine Timer-basierte Ausführung. Der Timer startet alle 5 Minuten den zugehörigen Oneshot-Service.

Timer-Unit (adminhelper-agent.timer)

[Unit]
Description=AdminHelper Agent Timer

[Timer]
OnBootSec=1min
OnUnitActiveSec=5min
RandomizedDelaySec=30
Persistent=true

[Install]
WantedBy=timers.target

Service-Unit (adminhelper-agent.service)

[Unit]
Description=AdminHelper Agent
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/adminhelper-agent run --once
StandardOutput=journal
StandardError=journal
  • RandomizedDelaySec=30: Verhindert gleichzeitige Anfragen vieler Agents.
  • Persistent=true: Verpasste Läufe werden beim nächsten Boot nachgeholt.
  • Type=oneshot: Service läuft einmal durch und beendet sich.

8) Windows-Integration

Unter Windows wird der Agent als nativer Windows-Service via sc.exe registriert.

Eigenschaft Wert
Service-Name AdminHelper-Agent
Binary C:\Program Files\AdminHelper\adminhelper-agent.exe run
Starttyp Automatisch
Recovery Neustart nach 60 Sekunden (3 Versuche) 
:: Service manuell registrieren (falls nicht via 'service install'):
sc.exe create AdminHelper-Agent ^
  binPath= "C:\Program Files\AdminHelper\adminhelper-agent.exe run" ^
  start= auto ^
  DisplayName= "AdminHelper Agent"

:: Recovery-Optionen setzen:
sc.exe failure AdminHelper-Agent reset= 86400 ^
  actions= restart/60000/restart/60000/restart/60000

9) Troubleshooting

SSL-Zertifikatsfehler

Bei selbstsignierten Zertifikaten das CA-Zertifikat mit --cacert /pfad/zur/ca.crt angeben. Alternativ --insecure für Testumgebungen verwenden.

Permission denied auf /etc/frp

Der Agent benötigt Root-Rechte für die Ersteinrichtung. Kommandos mit sudo ausführen.

frpc startet nicht

Status prüfen mit systemctl status frpc.service und journalctl -u frpc.service -n 50.

Monitoring-Push schlägt fehl

API_KEY und SERVER_ID in monitor.conf prüfen. Manueller Test: adminhelper-agent monitor push.

Diagnose-Tipp: Mit adminhelper-agent run --once kann ein einzelner Zyklus manuell ausgelöst und die Ausgabe direkt im Terminal geprüft werden.
