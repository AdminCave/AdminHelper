Monitoring

Das Monitoring-System besteht aus drei Komponenten: Checks (Konfigurationen, was wie oft geprüft wird), Collectors im Agent (führen Messungen durch) und VictoriaMetrics (speichert die Zeitreihen). Checks, Alert-Regeln und Templates werden seit v0.32.0 im Desktop-Client (Server-Modus) gepflegt – auf der fleet-weiten Monitoring-Seite und pro Server im Monitoring-Tab unter Infrastruktur. Die Charts lesen direkt von VictoriaMetrics via uPlot.

Check-Typen

Der Typ-Bezeichner (check_type) ist exakt der Wert, den der Desktop-Client im Dropdown anzeigt. Server-seitige Probes (ping, tcp, http) laufen direkt im Monitoring-Container, alle agent_*- und Plugin-Checks werten Push-Daten des Agents aus; disk_forecast liest die Metrik-Historie aus VictoriaMetrics.

Agent-Plugins werden automatisch erkannt

Die Plugin-Checks proxmox_backup, zfs_health und docker_health werten nur dann Daten aus, wenn der Agent das jeweilige Subsystem (Proxmox, ZFS, Docker) auf dem Host findet. Fehlt das Subsystem, bleibt der Check ohne Daten – kein Fehler. Die agent_*- und Plugin-Checks setzen den Go-Agent auf dem Ziel voraus; ping/tcp/http brauchen keinen Agent.

Check anlegen

  1. Desktop-Client (Server-Modus) → Monitoring → Neuer Check (oder pro Server: Infrastruktur → Server → Tab Monitoring)
  2. Typ wählen, Ziel (Server oder URL) auswählen
  3. Parameter ausfüllen – typabhängig, z. B. bei http: URL, erwarteter Status-Code, Timeout
  4. Intervall, Schweregrad (severity) und consecutive_fails setzen – die Schwellwerte selbst stehen in der typspezifischen Konfiguration (nächster Abschnitt)

Schwellwerte & Status

Jeder Check liefert einen der vier Status ok, warning, critical oder unknown. Die Schwellwerte dafür liegen in der typspezifischen Check-Konfiguration – es gibt kein globales Threshold-Feld. Die Konfiguration wird strikt validiert: unbekannte Schlüssel und Werte außerhalb der erlaubten Grenzen werden beim Speichern abgelehnt (statt still zu unknown zu führen).

Zusätzlich hat jeder Check zwei Dämpfungs-Parameter: severity (wie der Check eingestuft wird) und consecutive_fails (wie viele Fehl-Messungen in Folge nötig sind, bevor der Status kippt – Anti-Flapping; bei 5-Minuten-Intervall bedeutet 3 also ~15 Minuten Bestehen der Bedingung).

agent_resourcescpu_warn/cpu_crit (80/95), memory_warn/memory_crit (80/95), disk_warn/disk_crit (85/95), temp_warn/temp_crit (80/95 °C), hysteresis_pp (10)

Alert-Verhalten: Dämpfung & Unterdrückung

Hysterese

Der agent_resources-Check arbeitet pro Metrik mit einer Release-Schwelle: Steht z. B. die CPU eines Servers auf warning, sinken ihre Schwellen um hysteresis_pp (Default 10 Prozentpunkte), bis die Metrik wieder frei ist. Ein Server, der um die 90-%-Marke pendelt, erzeugt so eine Meldung statt einer Flapping-Serie – erst der Rückgang unter 80 % gilt als Entwarnung.

unknown benachrichtigt nie

Übergänge nach unknown (fehlende Agent-Daten, unerreichbare Datenquelle, unvollständige Konfiguration) lösen keine Benachrichtigung aus – der Zustand bleibt im Dashboard sichtbar, ist aber kein Vorfall. Ein echtes „Agent weg" meldet der agent_ping-Check als critical. unknown → ok wird als normale Recovery zugestellt.

Host-down-Unterdrückung

Steht der agent_ping eines Servers auf critical, bleiben alle anderen Checks dieses Servers stumm (Inhibition): ein Vorfall, eine Meldung. Sobald der Agent zurück ist, melden die Checks wieder normal.

Maintenance-Fenster

Pro Server lassen sich Wartungsfenster anlegen – einmalig (Start/Ende) oder wöchentlich (Wochentage + Zeitfenster in einer IANA-Zeitzone, z. B. Europe/Berlin; Sommer-/Winterzeit wird korrekt berücksichtigt). Verwaltet werden sie im Server-Monitoring-Tab (Infrastruktur → Server → Monitoring); aktive Fenster zeigt die Monitoring-Übersicht als Badge. Es gilt collect-but-mute: Checks laufen weiter und schreiben Metriken/Status, nur Benachrichtigungen (Alert-Regeln und Notification-Hub) sind unterdrückt.

Keine Nachmeldung nach Fenster-Ende

Der Alerter ist übergangsbasiert: Eine Status-Transition, die vollständig innerhalb des Fensters passiert (z. B. ok → critical → ok während der Wartung), wird nach Fenster-Ende nicht nachgemeldet. Steht der Check am Fenster-Ende noch auf einem Problem-Status, kommt die Meldung beim nächsten Übergang.

SMART-Monitoring

Der Agent erkennt automatisch SATA-, SAS- und NVMe-Festplatten (smartctl --scan-open). Für NVMe werden die Critical-Warning-Bits einzeln dekodiert:

BitBedeutung
0Spare-Capacity kritisch niedrig
1Temperatur außerhalb der Grenzwerte
2Device Reliability degradiert
3Read-Only-Mode
4Volatile Memory Backup fehlgeschlagen
5Persistent Memory Read-Only

Disk-Forecast (Voll-Prognose)

Der disk_forecast-Check errechnet aus der Füllraten-Historie in VictoriaMetrics (lineare Regression über window_hours, Default 24 h) pro Mountpoint die Zeit bis „Disk voll": warning bei voll in ≤ warn_hours (24), critical bei ≤ crit_hours (8). Stabile oder schrumpfende Belegung ist ok; unter min_history_hours (6) Historie bleibt der Check unknown – und benachrichtigt damit nie (siehe oben). Er warnt so deutlich vor den statischen disk_warn/disk_crit-Schwellen.

VictoriaMetrics

VictoriaMetrics speichert die Zeitreihen; die Aufbewahrung steuert die Compose-Variable VM_RETENTION (-retentionPeriod=${VM_RETENTION:-90d}, Default 90 Tage). Sie ist unabhängig vom Alert-Log (ALERT_LOG_RETENTION_DAYS, siehe unten) – die beiden Stellhebel syncen nicht. Pro Metrik werden vier Labels geschrieben: server, check, type, plus check-spezifische Labels wie mountpoint oder device. Direkter Zugriff (Debug): http://victoria:8428/api/v1/query?query=up.

Ingest-Authentifizierung

Der Agent sendet seine Metriken per POST /api/monitoring/agent/{server_id}/report an den AdminHelper-Server (Port 443, tunnelfrei) und authentifiziert mit seinem server-spezifischen X-API-Key (beim Provisioning vergeben). Der geteilte MONITOR_API_KEY ist davon getrennt: er sichert nur die interne Server→Monitoring-Proxy-Strecke (X-Internal-Key) und muss zwischen Server- und Monitoring-Container identisch sein.

Übersicht: Liste, Kacheln & Heartbeat

Die Monitoring-Seite des Desktop-Clients bietet zwei umschaltbare Ansichten (die Wahl wird gemerkt):

  • Liste: Server-Liste mit Dashboard – uPlot-Liniencharts, Checks-Übersicht, Alert-Regeln und Alert-Log. Zeitfenster wählbar: 1h, 6h, 24h, 7d, 30d; Klick auf eine Linie öffnet die Detail-Ansicht mit Rohdaten.
  • Kacheln: eine Kachel pro Server, eingefärbt nach dem schlechtesten Check-Status, mit Heartbeat-Balken – der Balken zeigt die Verfügbarkeit der letzten 24 h aus der agent_ping-Timeline (Server ohne agent_ping-Check zeigen keinen Balken).

Gibt es Server, für die noch kein einziger Check existiert, zeigt die Übersicht ein Banner („N Server ohne Monitoring") mit direktem Bulk-Zuweisungs-Dialog: Server vorausgewählt, Template wählbar (Standard-Templates zuerst). Die Ansicht aktualisiert sich automatisch alle 30 Sekunden.

Templates

Ein Monitoring-Template bündelt mehrere Check- und Alert-Definitionen und wird einem Server zugewiesen, statt jeden Check einzeln anzulegen. Beim Zuweisen werden aus den Definitionen echte Checks und Alert-Regeln für den Server erzeugt.

  1. Desktop-Client → Monitoring → Templates → Template anlegen, Check- und Alert-Definitionen hinzufügen
  2. Template einem Server zuweisen – AdminHelper erstellt alle enthaltenen Checks und Alert-Regeln
  3. Wird das Template später geändert, läuft ein Live-Sync: alle zugewiesenen Server werden automatisch aktualisiert (Checks erstellt, geändert oder gelöscht). Jede Definition trägt eine stabile def_id, über die der Sync zuordnet.

In Definitionen lassen sich Platzhalter verwenden, die beim Zuweisen pro Server ersetzt werden:

PlatzhalterErsetzt durch
{{hostname}}Hostname des Servers
{{server_name}}Anzeigename des Servers
{{server_id}}Interne Server-ID

Eine Zuweisung entfernen löscht die daraus erzeugten Checks und Alerts wieder. Ein Template zu löschen entfernt diese auf allen zugewiesenen Servern.

Standard-Templates

Beim ersten Start seedet der Monitoring-Dienst fünf englischsprachige Standard-Templates. Die Schwellwerte folgen dem Konsens gängiger Werkzeuge (Zabbix, Netdata, Checkmk) plus Backblaze-SMART-Statistiken:

TemplateInhalt
Linux Server — StandardAgent Heartbeat (stale nach 15 min ≈ 3 verpasste Pushes), System Resources (CPU 90/95, RAM 90/98, Disk 80/90 – mit consecutive_fails 3 ≈ 15 min Anti-Flap), Service Health (systemd, Modus auto), SMART Health, Disk Forecast
Windows Server — Standardwie Linux, ohne Service Health
Proxmox HostProxmox Backups (Backup-Alter < 26 h)
Docker HostDocker Health (exited/unhealthy Container)
ZFS StorageZFS Health (Pool-State + Kapazität 80/90)

Das Seeding ist einmalig pro Template: Wer ein Standard-Template löscht, umbenennt oder anpasst, behält diesen Stand – ein späterer Neustart stellt nichts wieder her und überschreibt nichts. Die Plugin-Templates (Proxmox/Docker/ZFS) sind bewusst getrennt: auf einem Host ohne das Subsystem stünde der Check dauerhaft auf unknown.

Templates zuweisen

Vier Wege führen zur Zuweisung:

  • Template-Modal (Monitoring → Templates): Server-Mehrfachauswahl zum Bulk-Zuweisen, Entfernen per Klick auf die Zuweisungs-Pille.
  • Server-Anlege-Dialog: optionales Template-Dropdown – beim Anlegen eines Servers direkt ein Standard-Template mitgeben (Opt-in, keine Automatik nach OS-Typ).
  • Server → Monitoring-Tab: Zuweisungen des einzelnen Servers ansehen, hinzufügen, entfernen.
  • Übersichts-Banner: der Bulk-Dialog für alle Server ohne Checks (siehe oben).

Tag-Zuweisung

Ein Template lässt sich zusätzlich an einen Server-Tag binden (im Template-Modal): Jeder Server, der den Tag trägt, bekommt das Template automatisch – auch Server, die den Tag erst später erhalten. Der Abgleich läuft sofort bei Server-Änderungen und zusätzlich alle 15 Minuten. Verliert ein Server den Tag, wird die via Tag entstandene Zuweisung (samt erzeugter Checks) wieder entfernt. Via Tag zugewiesene Server sind im Template-Modal als „via Tag" markiert; eine manuell angelegte Zuweisung bleibt manuell und wird vom Tag-Sync nie angefasst.

Benachrichtigungen & Alerting

Alert-Regeln feuern, wenn ein Check seinen Status wechselt (z. B. ok → critical). Jede Regel filtert optional nach severity und Server und versendet über einen Kanal.

KanalKonfiguration
webhookurl – AdminHelper sendet ein JSON-POST mit Check-Name, Typ, Server, Severity, altem/neuem Status und Klartext-Nachricht (Timeout 10 s, ohne Redirects, Erfolg bei HTTP < 300). Aus Sicherheitsgründen (SSRF) werden Ziele, die auf private/reservierte Adressen auflösen (z. B. 127.0.0.0/8, 10/8, 169.254.0.0/16), abgelehnt — für ein internes Webhook-Ziel den Empfänger über einen öffentlich auflösbaren Namen ansprechen.
emailrecipients (oder to, auch als CSV). SMTP standardmäßig aus den Umgebungsvariablen SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_FROM; pro Regel überschreibbar. Bei Port 465 wird implizites TLS (SMTPS) genutzt, bei jedem anderen Port STARTTLS; beide prüfen das Server-Zertifikat (kann kein TLS aufgebaut werden, schlägt der Versand fehl, statt Zugangsdaten im Klartext zu senden).

Cooldown: Nach einem erfolgreich versendeten Alert unterdrückt cooldown_minutes (Default 30) weitere Alerts derselben Regel für denselben Check.

Recovery: Geht ein Check zurück auf ok, wird eine Recovery-Benachrichtigung versendet – diese umgeht den Cooldown bewusst, damit die Entwarnung immer ankommt. Jeder Versand wird im Alert-Log mit Erfolg/Fehler protokolliert. Das Alert-Log wird täglich automatisch bereinigt (Retention konfigurierbar über ALERT_LOG_RETENTION_DAYS, Default 90 Tage).

Event-Hook: Bei Übergängen mit Schweregrad critical feuert der Server zusätzlich den Event-Hook alert.triggered (Payload: server_id, severity, title, message). Details zur Hook-Konfiguration: Hooks & Events.

Persönliche Benachrichtigungen (Notification-Hub)

Unabhängig von den regelbasierten Alert-Kanälen oben verteilt der Notification-Hub Ereignisse pro Benutzer: Jeder Nutzer entscheidet selbst, worüber und wie er benachrichtigt wird. Während die obigen webhook/email-Regeln an feste Ziele senden (z. B. einen Team-Webhook), richtet sich der Hub an die Personen, die einem Server zugeordnet sind.

Was wird zugestellt

  • Glocke im Desktop-Client (Kopfzeile): ein persönlicher Feed mit Ungelesen-Zähler, der sich in Echtzeit aktualisiert (Server-Push, solange eine Server-Sitzung aktiv ist) — eine neue Benachrichtigung erscheint sofort, ohne auf den nächsten Abruf zu warten. Das Öffnen markiert die Einträge als gelesen.
  • Desktop-Benachrichtigungen (native OS-Hinweise): optional, unter Einstellungen → Benachrichtigungen zu aktivieren (fragt einmalig die System-Berechtigung an).
  • E-Mail: zusätzlich, wenn in einer Regel aktiviert und eine Kontakt-E-Mail hinterlegt ist.

Eigene Regeln einstellen

Unter Einstellungen → Benachrichtigungen im Desktop-Client legt jeder Nutzer eine Kontakt-E-Mail und beliebig viele Regeln an. Jede Regel besteht aus:

FeldBedeutung
GeltungsbereichAlle Server (alle, die man sehen darf), Tag (alle Server mit diesem Tag) oder Einzelner Server.
Mindest-SchweregradAb welcher Stufe benachrichtigt wird: info, warning oder critical.
KanalDie Glocke erhält jeden Treffer; E-Mail ist optional zuschaltbar. (Telegram ist im Datenmodell vorbereitet, aber noch nicht in der Oberfläche freigeschaltet.)

Mehrere Regeln eines Nutzers werden zusammengeführt – ein Ereignis erzeugt genau einen Feed-Eintrag, die Kanäle sind die Vereinigung aller passenden Regeln.

Start-Regel für Admins: Neu angelegte Admin-Benutzer erhalten automatisch eine Regel Alle Server / Mindest-Schweregrad warning / nur Glocke – so erreicht auf einer frischen Installation jeder Alert mindestens die Admins. Die Regel ist normal editier- und löschbar; bestehende Nutzer bekommen nachträglich keine.

Was löst eine Benachrichtigung aus

  • Monitoring: jeder Statuswechsel eines Checks (z. B. ok → critical und die Entwarnung zurück). Der Schweregrad eines Übergangs ist der schlimmere der beiden Zustände, damit auch die Entwarnung Abonnenten mit Schwelle „warning" erreicht. Übergänge nach unknown, Wartungsfenster und Host-down-Unterdrückung gelten auch hier (siehe Alert-Verhalten).
  • Sicherheit: ein neuer Admin-Benutzer wurde angelegt.
  • Lifecycle: ein Server wurde aus dem Inventar entfernt, ein FRP-Tunnel wurde angelegt.
Nur was man sehen darf

Der Hub stellt einem Nutzer nur Ereignisse zu Servern zu, die er sehen darf – als Admin alle, sonst nur die ihm zugewiesenen (siehe Benutzer & Zugriff). „Alle Server" bedeutet entsprechend „alle für mich sichtbaren Server", nicht das gesamte Inventar.

E-Mail-Versand: ausgehende E-Mails werden über einen externen SMTP-Relay zugestellt (SMTP_HOST/SMTP_PORT/SMTP_USER/SMTP_PASSWORD/SMTP_FROM, siehe Betrieb & Konfiguration). Der Versand läuft asynchron über eine Outbox mit automatischem Wiederholversuch (linearer Backoff bis NOTIFICATION_MAX_ATTEMPTS, Default 5); ist kein SMTP-Relay konfiguriert, bleibt die Glocke trotzdem voll funktionsfähig.
