Monitoring

Define checks, set thresholds, wire alerts into hooks. Since v0.32.0 checks, alert rules, and templates are maintained in the desktop client (server mode) – on the fleet-wide Monitoring page and per server in the Monitoring tab under Infrastructure. The actual evaluation happens in the separate monitoring service, backed by VictoriaMetrics for metric storage (configurable retention, default 90 days).

Check Types

The type identifier (check_type) is exactly the value the desktop client shows in its dropdown. Server-side probes (ping, tcp, http) run inside the monitoring container; all agent_* and plugin checks evaluate data pushed by the agent; disk_forecast reads the metric history from VictoriaMetrics.

Agent plugins are auto-detected

The plugin checks proxmox_backup, zfs_health and docker_health only evaluate data when the agent finds the respective subsystem (Proxmox, ZFS, Docker) on the host. If the subsystem is absent, the check simply has no data – that is not an error. The agent_* and plugin checks require the Go agent on the target; ping/tcp/http need no agent.

Creating a check

  1. Desktop client (server mode) → Monitoring → New check (or per server: Infrastructure → Server → Monitoring tab)
  2. Pick the type, choose the target (server or URL)
  3. Fill in the parameters – type-specific, e.g. for http: URL, expected status code, timeout
  4. Set the interval, severity (severity) and consecutive_fails – the thresholds themselves live in the type-specific configuration (next section)

Thresholds & status

Every check evaluates to one of four states: ok, warning, critical or unknown. The thresholds live in the type-specific check configuration – there is no global threshold field. The configuration is validated strictly: unknown keys and out-of-bounds values are rejected at save time (instead of silently degrading to unknown).

In addition, every check carries two damping parameters: severity (how the check is classified) and consecutive_fails (how many failing measurements in a row are needed before the status flips – anti-flapping; at a 5-minute interval a value of 3 means the condition must hold for ~15 minutes).

Alert behavior: damping & suppression

Hysteresis

The agent_resources check applies a per-metric release threshold: once a server's CPU sits at warning, its thresholds drop by hysteresis_pp (default 10 percentage points) until the metric clears. A server hovering around the 90 % mark produces one notification instead of a flapping series – only dropping below 80 % counts as recovery.

unknown never notifies

Transitions into unknown (missing agent data, unreachable data source, incomplete configuration) never trigger a notification – the state stays visible on the dashboard, but it is not an incident. A real "agent gone" is reported by the agent_ping check going critical. unknown → ok is delivered as a normal recovery.

Host-down suppression

While a server's agent_ping stands at critical, every other check of that server stays silent (inhibition): one incident, one notification. Once the agent is back, checks report normally again.

Maintenance windows

Maintenance windows are defined per server – one-off (start/end) or weekly (weekdays + time window in an IANA time zone, e.g. Europe/Berlin; daylight-saving transitions are handled correctly). They are managed in the server's Monitoring tab (Infrastructure → Server → Monitoring); active windows show as a badge on the monitoring overview. The semantics are collect-but-mute: checks keep running and keep writing metrics/status, only notifications (alert rules and the notification hub) are suppressed.

No catch-up after the window ends

The alerter is transition-based: a status transition that happens entirely inside the window (e.g. ok → critical → ok during maintenance) is not reported retroactively after the window ends. If the check still shows a problem state when the window closes, the notification arrives with the next transition.

SMART monitoring

The agent auto-detects SATA, SAS and NVMe disks (smartctl --scan-open). For NVMe the critical-warning bits are decoded individually:

BitMeaning
0Available spare below threshold
1Temperature above critical
2NVM subsystem reliability degraded
3Media placed in read-only mode
4Volatile memory backup device failed
5Persistent memory region unreliable

Disk forecast (time-until-full)

The disk_forecast check computes, per mountpoint, the time until "disk full" from the fill-rate history in VictoriaMetrics (linear regression over window_hours, default 24 h): warning when full within warn_hours (24), critical within crit_hours (8). Stable or shrinking usage is ok; with less than min_history_hours (6) of history the check stays unknown – and therefore never notifies (see above). It warns well before the static disk_warn/disk_crit thresholds fire.

VictoriaMetrics

VictoriaMetrics stores the time series; retention is controlled by the compose variable VM_RETENTION (-retentionPeriod=${VM_RETENTION:-90d}, default 90 days). It is independent of the alert log (ALERT_LOG_RETENTION_DAYS, see below) – the two knobs do not sync. Four labels are written per metric: server, check, type, plus check-specific labels like mountpoint or device. Direct access (debug): http://victoria:8428/api/v1/query?query=up.

Ingest authentication

The agent sends its metrics via POST /api/monitoring/agent/{server_id}/report to the AdminHelper server (port 443, tunnel-free) and authenticates with its server-specific X-API-Key (issued at provisioning). The shared MONITOR_API_KEY is separate from that: it only secures the internal server→monitoring proxy leg (X-Internal-Key) and must be identical between the server and monitoring containers.

Overview: list, tiles & heartbeat

The desktop client's Monitoring page offers two switchable views (the choice is remembered):

  • List: server list with dashboard – uPlot line charts, checks overview, alert rules and the alert log. Selectable time windows: 1h, 6h, 24h, 7d, 30d; clicking a line opens the detail view with raw data.
  • Tiles: one tile per server, colored by the worst check status, with a heartbeat bar – the bar shows the availability of the last 24 h from the agent_ping timeline (servers without an agent_ping check show no bar).

If servers exist that have no check at all, the overview shows a banner ("N servers without monitoring") with a direct bulk-assign dialog: servers preselected, template selectable (standard templates first). The view refreshes automatically every 30 seconds.

Templates

A monitoring template bundles several check and alert definitions and is assigned to a server instead of creating each check by hand. On assignment the definitions are materialized into real checks and alert rules for that server.

  1. Desktop client → Monitoring → Templates → create a template, add check and alert definitions
  2. Assign the template to a server – AdminHelper creates all contained checks and alert rules
  3. When the template is edited later, a live sync runs: every assigned server is updated automatically (checks created, updated or deleted). Each definition carries a stable def_id the sync uses to match.

Definitions may use placeholders that are substituted per server at assignment time:

PlaceholderReplaced with
{{hostname}}Server hostname
{{server_name}}Server display name
{{server_id}}Internal server ID

Removing an assignment deletes the checks and alerts it created. Deleting a template removes them on all assigned servers.

Standard templates

On first startup the monitoring service seeds five standard templates (English names). The thresholds distill the consensus of common tools (Zabbix, Netdata, Checkmk) plus Backblaze SMART statistics:

TemplateContents
Linux Server — StandardAgent Heartbeat (stale after 15 min ≈ 3 missed pushes), System Resources (CPU 90/95, RAM 90/98, disk 80/90 – with consecutive_fails 3 ≈ 15 min anti-flap), Service Health (systemd, auto mode), SMART Health, Disk Forecast
Windows Server — Standardlike Linux, without Service Health
Proxmox HostProxmox Backups (backup age < 26 h)
Docker HostDocker Health (exited/unhealthy containers)
ZFS StorageZFS Health (pool state + capacity 80/90)

Seeding is one-time per template: if you delete, rename or tune a standard template, that state sticks – a later restart restores nothing and overwrites nothing. The plugin templates (Proxmox/Docker/ZFS) are deliberately separate: on a host without the subsystem the check would sit at unknown forever.

Assigning templates

There are four ways to assign:

  • Template modal (Monitoring → Templates): server multi-select for bulk assignment, removal via the assignment pill.
  • Server-create dialog: optional template dropdown – hand a standard template to a server right at creation (opt-in, no automatic assignment by OS type).
  • Server → Monitoring tab: view, add and remove the assignments of a single server.
  • Overview banner: the bulk dialog for all servers without checks (see above).

Tag assignment

A template can additionally be bound to a server tag (in the template modal): every server carrying the tag gets the template automatically – including servers that receive the tag later. The sync runs immediately on server changes and additionally every 15 minutes. When a server loses the tag, the tag-created assignment (and its materialized checks) is removed again. Tag-assigned servers are marked "via tag" in the template modal; a manually created assignment stays manual and is never touched by the tag sync.

Notifications & alerting

Alert rules fire when a check changes status (e.g. ok → critical). Each rule optionally filters by severity and server and dispatches over one channel.

ChannelConfiguration
webhookurl – AdminHelper sends a JSON POST with check name, type, server, severity, old/new status and a plain-text message (10 s timeout, no redirects, success on HTTP < 300). For security (SSRF), targets that resolve to private/reserved addresses (e.g. 127.0.0.0/8, 10/8, 169.254.0.0/16) are rejected — reach an internal webhook receiver via a publicly resolvable name.
emailrecipients (or to, also as CSV). SMTP defaults from the environment variables SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD, SMTP_FROM; overridable per rule. Port 465 uses implicit TLS (SMTPS); every other port uses STARTTLS. Both verify the server certificate (if TLS cannot be established the send fails rather than sending credentials in the clear).

Cooldown: After a successfully sent alert, cooldown_minutes (default 30) suppresses further alerts from the same rule for the same check.

Recovery: When a check returns to ok, a recovery notification is sent – it deliberately bypasses the cooldown so the all-clear always gets through. Every dispatch is recorded in the alert log with success/error. The alert log is pruned automatically every day (retention configurable via ALERT_LOG_RETENTION_DAYS, default 90 days).

Event hook: On transitions into critical the server additionally fires the alert.triggered event hook (payload: server_id, severity, title, message). Hook configuration details: Hooks & Events.

Personal notifications (notification hub)

Independently of the rule-based alert channels above, the notification hub fans events out per user: each user decides what they are notified about and how. Where the webhook/email rules above send to fixed destinations (e.g. a team webhook), the hub targets the people assigned to a server.

What is delivered

  • Bell in the desktop client (header): a personal feed with an unread counter that refreshes in real time (server push, while a server session is active) — a new notification appears immediately, without waiting for the next poll. Opening it marks the entries as read.
  • Desktop notifications (native OS alerts): optional, enabled under Settings → Notifications (asks for the system permission once).
  • Email: additionally, when enabled in a rule and a contact email is set.

Configure your own rules

Under Settings → Notifications in the desktop client every user maintains a contact email and any number of rules. Each rule consists of:

FieldMeaning
ScopeAll servers (all the user may see), Tag (all servers with that tag) or Single server.
Minimum severityFrom which level on to notify: info, warning or critical.
ChannelThe bell receives every match; email can be enabled on top. (Telegram is prepared in the data model but not yet exposed in the UI.)

Multiple rules of one user are merged – one event produces exactly one feed entry, and the channels are the union of all matching rules.

Starter rule for admins: Newly created admin users automatically get a rule All servers / minimum severity warning / bell only – so on a fresh install every alert reaches at least the admins. The rule can be edited or deleted like any other; existing users do not receive one retroactively.

What triggers a notification

  • Monitoring: every status transition of a check (e.g. ok → critical and the recovery back). The severity of a transition is the worse of the two states, so the all-clear also reaches subscribers with a "warning" threshold. Transitions into unknown, maintenance windows and host-down suppression apply here too (see Alert behavior).
  • Security: a new admin user was created.
  • Lifecycle: a server was removed from the inventory, an FRP tunnel was created.
Only what you may see

The hub only delivers events for servers the user is allowed to see – all of them for admins, otherwise only the assigned ones (see Users & Access). "All servers" accordingly means "all servers visible to me", not the whole inventory.

Email delivery: outgoing emails are delivered via an external SMTP relay (SMTP_HOST/SMTP_PORT/SMTP_USER/SMTP_PASSWORD/SMTP_FROM, see Operation & Configuration). Delivery runs asynchronously through an outbox with automatic retry (linear backoff up to NOTIFICATION_MAX_ATTEMPTS, default 5); without a configured SMTP relay the bell remains fully functional.
