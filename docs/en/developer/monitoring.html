Monitoring Stack

Monitoring is three layers cooperating: SRM server defines checks and aggregates alerts, the separate monitoring/ service executes check logic and writes metrics, VictoriaMetrics stores the time-series data.

Data Flow

flow
[Agent]                [Monitoring service]       [VictoriaMetrics]
    │                         │                            │
    │ POST /ingest ─ metrics ▶│                            │
    │                         │ /api/v1/import/prometheus ▶│
    │                         │                            │
                              │ ◀ pull config from /api/monitoring
                              │
   [SRM server] ─ schedule ─▶ APScheduler triggers checks ─▶ [Monitoring service]
                              │
                              │ SRM ◀ query charts ◀ uPlot (web panel)

SRM Server Components

  • server/app/monitoring/checks.py – Pydantic schemas per check type
  • server/app/monitoring/scheduler.py – APScheduler integration
  • server/app/monitoring/proxy.py – SSRF-safe proxy to VictoriaMetrics
  • server/app/monitoring/alerts.py – threshold evaluation + hook trigger

Monitoring Service

Standalone FastAPI app under monitoring/. Listens on port 8080 (container-internal), 8480 is exposed for agent ingest. Authenticates via MONITOR_API_KEY.

Check-Type Schemas

Each check type has a Pydantic schema for config and an executor. Example http:

python
class HttpCheckConfig(BaseModel):
    url: HttpUrl
    method: Literal["GET", "HEAD", "POST"] = "GET"
    expected_codes: list[int] = [200]
    timeout_sec: float = 5.0
    verify_tls: bool = True
    headers: dict[str, str] | None = None

The executor calls httpx.AsyncClient, measures latency, builds a CheckResult and pushes metrics to VictoriaMetrics.

SSRF Protection

The monitoring proxy inside the server forwards VictoriaMetrics queries from the web panel, but the query URL is strictly allow-listed: /api/v1/query and /api/v1/query_range. Raw host access is blocked so internal services cannot be reached.

Ingest validation

The ingest endpoint enforces that samples from an agent belong to the agent's own server ID. A compromised agent key cannot forge metrics for other servers.

Local Debugging

bash
# Run a single check with debug output to stdout
docker compose exec monitoring python -m app.cli run-check http <id>

# Query VictoriaMetrics directly
docker compose exec monitoring curl -s \
    "http://victoria:8428/api/v1/query?query=cpu_load{server=\"srv1\"}"
