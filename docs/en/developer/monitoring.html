Monitoring Stack

Monitoring is three layers cooperating: the AdminHelper server acts as a reverse proxy and forwards UI queries, the separate monitoring/ service holds checks/alerts/templates and runs scheduler-based pull checks, VictoriaMetrics stores the time-series data (retention configurable via VM_RETENTION, default 90 days).

Data Flow

flow
[Agent]                                [Monitoring service]            [VictoriaMetrics]
   |                                          |                                |
   | POST /agent/{server_id}/report --------->|  agent_resources, service_     |
   |   (X-API-Key)                            |  process, docker_health,        |
   |                                          |  proxmox_backup, zfs_health,    |
   |                                          |  smart_health -> evaluate +     |
   |                                          |  format_line() ---------------->|
   |                                          |                                |
                                              |  APScheduler -> http / ping /  |
                                              |  tcp / agent_ping ------------>|
                                              |
[Desktop client] -> /api/monitoring/* (admin) -> AdminHelper server -> /monitor/* (X-Internal-Key) -> Monitoring service

AdminHelper side: monitoring_proxy

There is no dedicated monitoring module on the AdminHelper server anymore; everything goes through apps/server/app/modules/monitoring_proxy.py:

  • POST /api/monitoring/agent/{server_id}/report – public (X-API-Key), forwarded 1:1 to the service
  • GET/POST/PUT/DELETE /api/monitoring/{path:path} – admin-only, internally augmented with X-Internal-Key: $MONITOR_API_KEY
  • SSRF guard: allowed path prefixes are checks, alerts, log, metrics, status, templates, maintenance, agent; path traversal (..) is blocked
  • Deleting a server in AdminHelper fires DELETE /servers/{id}/cleanup at the monitoring service – orphaned checks/alerts are pruned automatically

Push checks vs. pull checks

There are two classes of checks – what matters is who gathers the data:

ModeCheck typesTrigger
Push (agent)agent_resources, service_process, docker_health, proxmox_backup, zfs_health, smart_healthOn the next agent push (5-minute timer); APScheduler is not involved.
Pull (service)http, ping, tcp, agent_ping, disk_forecastAPScheduler inside the monitoring service, by a fixed interval.

Both paths converge in execute_check() → status update in MonitorState → format_line() → VictoriaMetrics write → process_alert() when fail_count ≥ consecutive_fails.

APScheduler (scheduler.py) runs with a thread pool of 30 workers and misfire_grace_time=30 s – runs that start late are caught up within 30 s instead of being dropped. A daily cleanup job additionally prunes monitor_alert_log (retention via ALERT_LOG_RETENTION_DAYS, default 90 days), and the tag sync runs every 15 minutes (see below).

Push payload (agent → service)

The Go agent assembles the report in apps/agent/internal/monitor/report.go (BuildReport) and sends it as JSON via POST {url}/agent/{server_id}/report with the headers Content-Type: application/json and X-API-Key (PushReport). The server handler in routers/agent.py accepts the JSON as a free-form dict. Structure:

json
{
  "report_version": 2,
  "timestamp": "2026-06-04T10:00:00Z",
  "uptime_seconds": 864000,
  "resources": {
    "cpu_percent": 45.2,
    "memory_percent": 61.0,
    "memory_total_mb": 16000,
    "memory_used_mb": 9760,
    "load_1m": 0.8, "load_5m": 0.6, "load_15m": 0.5,
    "disks": [
      { "mount": "/", "fstype": "ext4", "percent": 72.0,
        "total_gb": 100, "used_gb": 72 }
    ],
    "temperatures": [
      { "sensor": "Package id 0", "temp_c": 54, "high": 80, "critical": 100 }
    ]
  },
  "systemd": { "all_services": [ /* unit, active_state, enabled_state */ ],
               "failed": [], "watched": [] },
  "services": [ { "name": "nginx", "running": true } ],
  "docker":  { /* plugin: docker_health */ },
  "proxmox": { /* plugin: proxmox_backup */ },
  "zfs":     { /* plugin: zfs_health */ },
  "smart":   [ { "device": "/dev/sda", "temp_c": 38,
                 "reallocated_sectors": 0, "pending_sectors": 0 } ]
}

Fields are optional: docker/proxmox/zfs/smart appear only when the agent detects the respective subsystem; VMs without sensors send no temperatures. services is a legacy key alongside systemd. The handler iterates over resources (CPU/RAM/load/disk/temp), uptime_seconds and smart, writes VictoriaMetrics lines from them (pseudo filesystems in EXCLUDED_FSTYPES are skipped), then evaluates the push checks configured for the server against the report. Response: {"status":"ok","metricsWritten":N,"checksUpdated":M}.

systemd.all_services is additionally throttled: the agent sends the full inventory only when its hash has changed or the last full send is more than 1 h ago. When the key is missing, the ServiceProcessChecker falls back to the always-sent keys failed/enabled_inactive – a missing key ≠ an empty list (an empty list means: genuinely no services).

Checkers at a glance

Each checker returns a triple (status, message, metrics) with status ∈ {ok, warning, critical, unknown}. Pull checkers expose a run(config) method; agent/push checkers evaluate the push report directly in evaluate(config, report).

CheckerType(s)What it does
http.pyhttpHTTP/HTTPS request via httpx (method configurable, follows redirects). Compares the status code against expected_status, optionally searches for search_string in the body, measures response time. SSRF guard: URLs that resolve to private/reserved IP ranges are rejected.
ping.pypingICMP echo via ping -c 1 -W <timeout> as a subprocess. The target is validated against an allowlist regex (hostnames/IPs); parses the RTT from the output.
tcp.pytcpTCP connect via socket.create_connection((target, port)) with timeout. ok on a successful connect (with connect time), critical on timeout/refused.
agent.pyagent_ping, agent_resources, service_processAgentPingChecker checks (scheduler-driven) the age of the last push against stale_minutes. AgentResourcesChecker evaluates CPU/RAM/disk/temp against warn/crit thresholds. ServiceProcessChecker checks services – mode auto (failed + enabled-but-inactive systemd units, with an ignore list) or list (explicitly named services). The resources checker applies per-metric hysteresis (hysteresis_pp, release = entry − 10 pp); push timestamps are persisted by the push handler (routers/agent.py) in monitor_agent_liveness and hydrated by the lifespan at startup – no false-alert storm after a monitoring restart.
forecast.pydisk_forecastPull checker: query_range over monitor_agent_disk_percent_value{server_id="…"} (window window_hours, step 10 m), least-squares slope per mount → hours_left = (100 − last) / rate. Rates ≤ 0.01 pp/h count as stable (ok); too little history → unknown. server_id is escaped with escape_label_value().
smart.pysmart_healthEvaluates SMART data from the push (ATA HDD/SSD and NVMe). Checks reallocated/pending sectors, NVMe critical_warning bits and temperatures against type-specific thresholds, among others. A missing smart key (VM or no smartctl) is ok, not an error.

The plugin checkers proxmox_backup, zfs_health and docker_health (in checkers/plugins.py) are likewise push-evaluated.

Auth modes

Both modes are implemented in core/auth.py as FastAPI dependencies:

  • require_agent (agent push): reads header X-API-Key. If the value matches the internal key exactly, it counts as __internal__ (access to all servers). Otherwise the key is SHA-256-hashed via MonitorAgentKey.hash_key() and compared against the hashed_key column in monitor_agent_keys – on a hit the dependency returns the associated server_id. A clear-text key is never stored; only the hash lives in the DB.
  • require_internal (AdminHelper ↔ service): reads header X-Internal-Key and compares it against the internal key. Its source is the env var MONITOR_API_KEY; if absent, a random key is generated at startup and persisted to $DATA_DIR/.api_key (core/config.py).

Agent key: issuance & server binding

Agent keys are issued per server (routers/admin.py):

  • POST /agent-keys/{server_id} creates a raw key with secrets.token_urlsafe(48), stores only its SHA-256 hash, and returns the raw key once in the response. If a key already exists for the server, it is deleted first (rotation = DELETE + POST). server_id is unique in the table, so exactly one key per server.
  • Both endpoints (POST/DELETE /agent-keys/{server_id}) are themselves guarded by require_internal – only the AdminHelper server may manage keys.
  • The push handler (routers/agent.py) additionally checks that the server_id derived from the key matches the server_id in the path; otherwise 403. A valid key for server A therefore cannot submit metrics for server B (__internal__ excepted).

Templates, seeding & tag sync

MonitorTemplate bundles check and alert definitions as JSON with a stable def_id per definition. POST /templates/{id}/assign/{server_id} rolls out the template; template_sync.py reconciles changes so server assignments stay drift-free.

Built-in seeding (tombstone)

At startup the lifespan seeds five standard templates from builtin_templates.py (marked via builtin_slug). The tombstone in monitor_seed_state guarantees: each slug is seeded at most once – a user-deleted or user-edited built-in is never restored or overwritten by later startups; new built-ins of later releases seed only themselves. The definitions pass the same schema boundary as user templates (test tests/test_seed.py).

Tag sync (server ↔ monitoring)

Tags live in the server DB (Server.tags); monitoring only knows server_id strings. tag_sync.py pulls the inventory via GET /api/internal/servers on the AdminHelper server (service-to-service, X-Internal-Key) and materializes the server assignments for every MonitorTemplateTagAssignment with source='tag'. Triggers: every 15 minutes via the scheduler and immediately as a best-effort nudge – the AdminHelper server POSTs to /templates/tag-sync after server create/update(tags/name/hostname)/delete. When a server loses the tag, the tag assignment and its materialized checks are removed; source='manual' assignments are never touched by the sync. An unreachable server API (None) is strictly distinguished from an empty inventory ([]) – an outage deletes no assignments.

Alert pipeline: guards & suppression

Order in process_alert() (applies to rule dispatch and the hub emit, since _emit_to_hub sits behind the same guards):

  1. old == new → no alert (real transitions only).
  2. Transition into unknown → never notify (unknown policy); unknown → ok is a normal recovery.
  3. Active maintenance window of the server → collect-but-mute: status/metrics keep flowing, the notification is suppressed, no alert-log row. is_in_maintenance() evaluates weekly windows in the configured IANA time zone (zoneinfo, DST-correct; including day wrap across midnight).
  4. Host-down inhibition: while the server's agent_ping stands at critical, all other checks of that server stay silent; agent_ping itself always alerts.

Maintenance CRUD lives in routers/maintenance.py (/maintenance, require_internal; proxy prefix maintenance).

VictoriaMetrics

External container (see docker-compose.yml), retention -retentionPeriod=${VM_RETENTION:-90d} – independent of ALERT_LOG_RETENTION_DAYS (alert log in Postgres). The service writes in InfluxDB line protocol via core/victoria.py:

bash
monitor_agent_cpu_percent,server_id=srv123 value=45.2 1683000000

Reads use the standard Prometheus endpoints /api/v1/query and /api/v1/query_range. uPlot in the desktop client renders the charts. The UI chart queries are never raw: GET /checks/{id}/metrics builds the PromQL query server-side from a per-check-type metric allowlist (CHECK_TYPE_METRICS) and escapes label values – free-text queries from the UI never reach VictoriaMetrics (SSRF/injection guard).

Ingest validation

The push endpoint enforces that the API key really belongs to the supplied server_id. A compromised agent key cannot forge metrics for other servers.
