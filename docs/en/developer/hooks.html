Hooks & Events

The hook system is the primary outbound integration surface. Every relevant server event fires an event, registered scripts run in a sanitised subprocess with a hard timeout.

Architecture

tree
apps/server/app/modules/hooks/
├── router.py          /api/hooks CRUD
├── models.py          Hook
├── schemas.py         Pydantic schemas
├── scheduler.py       Schedule-hook cron + notification outbox
├── script_runner.py   Async subprocess spawn
└── script_worker.py   Child process: stdin=JSON, stdout/stderr capture

Hook Definition

json
{
  "name": "Slack alert",
  "hook_type": "event",
  "event_triggers": ["server.updated"],
  "script": "http_post('https://hooks.slack.com/services/T000/B000/XXX', json_data={'text': f'AdminHelper: {event_type}'})\nresult['notified'] = True"
}

The script field is Python (not shell), run by the worker via exec(). The event's context keys are available as variables (for event hooks event_type and event_data, for schedule triggered_at/last_run, for webhook payload/headers/params); output goes through the result dict and log(...). For security, headers is restricted to a safe allow-list (content-type, user-agent, x-hook-source) — client-controlled headers like Authorization, Cookie and X-Forwarded-* are never exposed to the script. The whole context is passed to the worker as JSON (json.dumps with default=str), so only JSON-native types survive — non-native values in event_data (datetime, Decimal, bytes, ORM objects) reach the script as their str() form, not the original type.

Environment Model

Hook scripts run in a dedicated Python subprocess with a reduced environment — this is crash/timeout isolation, not a security sandbox:

  • Reduced env: only PATH, PYTHONPATH, DATABASE_URL and DATA_DIR are passed through (ADMIN_PASSWORD, MONITOR_API_KEY, REDIS_URL are dropped). SECRET_KEY and the DB credentials stay reachable, however (via DATA_DIR/.secret_key resp. DATABASE_URL) — no secret jail.
  • CWD is the server directory (server_dir), not a temp directory.
  • Timeout via subprocess.run(timeout=…), default 30 s; on expiry the worker process is killed, not necessarily its child processes.
  • No output limit and no user switch: stdout/stderr are captured in full, and the worker runs as the same user as the server.
Not a full sandbox

The hook isolation is defense-in-depth, not a seccomp/namespace jail. Admins with hook-CRUD privileges can run arbitrary code on the server host. Grant hook access only to trusted roles.

Failure behaviour

If a hook fails (exit_code != 0 or timeout), the error is logged: event and schedule hooks via logger.exception in the server log, the webhook trigger additionally in the response body (success: false, error). There is currently no automatic retry and no hook.failed meta-event; a failed hook blocks neither other hooks nor the triggering request.

Script Runner Implementation

python
SCRIPT_TIMEOUT_SECONDS = 30

def run_hook_script(script, hook_type, context, timeout=SCRIPT_TIMEOUT_SECONDS):
    payload = json.dumps({"script": script, "context": context}, default=str)

    # Reduced env: drops ADMIN_PASSWORD, MONITOR_API_KEY, REDIS_URL. NOT a jail —
    # DATABASE_URL is deliberately inherited (hooks need DB), SECRET_KEY stays
    # reachable via DATA_DIR/.secret_key. Hooks = trusted code.
    server_dir = str(Path(__file__).parents[3])
    worker_env = {
        "PATH": os.environ.get("PATH", ""),
        "PYTHONPATH": server_dir,
        "DATABASE_URL": os.environ.get("DATABASE_URL", ""),
        "DATA_DIR": os.environ.get("DATA_DIR", ""),
    }
    try:
        proc = subprocess.run(
            [sys.executable, _WORKER_SCRIPT],  # runs the script via a Python worker
            input=payload, capture_output=True, text=True,
            timeout=timeout, cwd=server_dir, env=worker_env,
        )
    except subprocess.TimeoutExpired:
        return {"success": False, "error": f"Timeout nach {timeout}s"}
    return _parse_worker_output(proc)  # success/result/logs from stdout

Firing Custom Events

Internal code can emit events at any time:

python
from app.core.events import fire_event

fire_event("server.updated", {"id": server.id, "name": server.name})

fire_event is non-blocking: the dispatch runs in a thread pool, the call returns immediately. The HTTP response of the emitter does not wait on the hook.

Test Execution

Every hook can be tested from the web panel with a sample event: POST /api/hooks/{id}/test. The execution log shows stdout/stderr/exit_code in real time – handy for script debugging.
