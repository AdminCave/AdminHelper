API Reference

Complete reference for all AdminHelper Server API endpoints. Base URL: https://<server>:443/api

Authentication Connections

1) Authentication

The API supports two authentication methods:

JWT Bearer Token

Login via POST /api/auth/login returns an access_token (validity: 8 h) and a refresh_token (validity: 7 d). The access token is passed as a Bearer token in the header. 

# Login and obtain token
curl -X POST https://server:443/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "geheim"}'

# Use the token
curl https://server:443/api/connections \
  -H "Authorization: Bearer <access_token>"

API-Key

API keys are created via the web UI or the API. The key is passed in the X-API-Key header. Permissions: read or read_write. 

# Use an API key
curl https://server:443/api/connections \
  -H "X-API-Key: <api_key>"
Auth levels in the tables below:
  • Public — No token required
  • JWT — Bearer token required
  • Admin — Bearer token + admin flag
  • API-Key — API key header (read or read_write)
  • JWT|API-Key — Either accepted

2) Auth Endpoints

Auth — Login, Refresh, Logout
Method Path Auth Description
POST /api/auth/login Public Login, returns access and refresh tokens
POST /api/auth/refresh JWT Renew access token
POST /api/auth/logout JWT Revoke token (blacklist)
GET /api/auth/me JWT Get current user

12) Hooks

Hooks — Webhooks, events, and schedules
Method Path Auth Description
GET /api/hooks Admin List all hooks
POST /api/hooks Admin Create hook (webhook/event/schedule)
PUT /api/hooks/{id} Admin Update hook
DELETE /api/hooks/{id} Admin Delete hook
POST /api/hooks/{id}/trigger Admin Trigger hook manually
POST /api/hooks/incoming/{token} Public Webhook endpoint (via token)

Hook Types

  • Webhook: HTTP POST with token authentication, rate-limited (20/min).
  • Event: Triggered automatically on system events (connection.created, server.updated, frp.config.created etc.).
  • Schedule: Cron-based, e.g. 0 2 * * * = daily at 02:00.

15) Monitoring Proxy

The AdminHelper server forwards monitoring requests as a reverse proxy to the internal monitoring service. All paths below /api/monitoring/ are transparently proxied.

Monitoring — Proxy to monitoring service
Method Path Auth Description
* /api/monitoring/* Admin Proxy to monitoring service
